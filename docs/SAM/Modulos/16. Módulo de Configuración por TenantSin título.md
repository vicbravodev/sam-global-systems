Es el módulo encargado de personalizar el comportamiento funcional, operativo y visual de SAM para cada cliente, permitiendo adaptar reglas, umbrales, canales, políticas y preferencias según las necesidades específicas de cada tenant.

Este módulo proporciona la **capa de parametrización del producto**, haciendo posible que una misma plataforma opere de forma distinta para distintos clientes sin necesidad de cambiar código ni duplicar lógica.

---

## Responsabilidad

Administrar:

- configuraciones funcionales por tenant
- overrides de reglas y políticas del sistema
- preferencias de notificación y escalamiento
- perfiles de comportamiento para AI
- branding y personalización visual
- horarios operativos y ventanas de atención
- parámetros que modulan la operación del sistema por cliente

---

## Qué resuelve

- personalización del sistema por cliente
- adaptación de reglas y umbrales operativos
- configuración de canales y políticas de notificación
- control de escalamiento por tenant
- ajuste del comportamiento de AI
- branding y experiencia visual diferenciada
- soporte a distintas operaciones sin forks del producto
- gobernanza de parámetros sin tocar código

---

## Alcance del módulo

Este módulo gobierna todo lo relacionado con **cómo debe comportarse SAM para un tenant específico**.

Su propósito es responder preguntas como:

- ¿qué umbral de severidad aplica a este cliente?
- ¿qué canales de notificación están habilitados?
- ¿qué reglas están activas o modificadas?
- ¿cómo deben escalarse ciertos eventos?
- ¿qué tono, prompt o perfil debe usar la AI?
- ¿qué branding debe mostrarse?
- ¿cuáles son los horarios operativos del cliente?

Este módulo no define la existencia del tenant ni sus features base, pero sí define la **configuración operativa concreta** que ese tenant aplica dentro del sistema.

# Reglas de negocio

## Configuración aislada por tenant

- toda configuración debe pertenecer a un tenant
- ningún tenant debe poder afectar la configuración de otro
- debe existir posibilidad de defaults globales con override local

## Override controlado

- los tenants pueden sobrescribir ciertos parámetros, pero no todo
- algunas reglas o políticas pueden ser no editables por seguridad o producto

## Versionado

- configuraciones relevantes deben poder versionarse
- cambios importantes deben ser auditables
- idealmente debe existir rollback en ciertos casos

## Consistencia operativa

- la configuración activa debe ser resoluble de forma determinística
- si hay conflicto entre default y override, debe existir precedencia clara

## Efecto transversal

- la configuración del tenant impacta múltiples módulos:
    - decisiones
    - AI
    - notificaciones
    - escalamiento
    - branding
    - horarios

## Seguridad y permisos

- no cualquier usuario debe editar configuraciones críticas
- debe haber control por rol y auditoría de cambios

# Precedencia recomendada de configuración

Yo usaría esta jerarquía:

1. defaults globales del sistema
2. configuración por plan o feature
3. configuración específica del tenant
4. override puntual por tenant
5. override temporal/manual si existiera

Eso hace más claro cómo resolver conflictos.

---

# Flujos principales

## 1. Resolución de configuración operativa

1. Un módulo necesita un parámetro
2. Se consultan defaults globales
3. Se aplican settings del tenant
4. Se aplican overrides si existen
5. Se entrega configuración efectiva

---

## 2. Cambio de política del tenant

1. Usuario autorizado edita configuración
2. Se valida consistencia
3. Se guarda nuevo valor
4. Se audita el cambio
5. Se actualiza versión efectiva

---

## 3. Cambio de perfil AI

1. Tenant define nueva política de AI
2. Se actualiza `TenantAIProfile`
3. Nuevas evaluaciones usan el perfil actualizado
4. Reevaluaciones futuras pueden usar versión nueva o mantener referencia histórica

---

## 4. Configuración de notificaciones

1. Tenant define canales y fallback
2. Se guarda `TenantNotificationPolicy`
3. El módulo de notificaciones consulta esa política
4. Se resuelven canales efectivos al enviar

---

## 5. Configuración de horarios

1. Tenant define horario operativo y días especiales
2. Se guarda `TenantScheduleProfile`
3. Módulos de contexto, decisiones y notificaciones lo usan
4. Cambia comportamiento dentro/fuera de horario

---

# Servicios principales

## ResolveTenantSetting

- obtiene la configuración efectiva de un parámetro
- aplica jerarquía de defaults y overrides

## UpdateTenantSetting

- crea o actualiza una configuración del tenant
- valida formato y permisos

## ApplyTenantRuleOverrides

- resuelve overrides sobre reglas base
- modifica comportamiento sin duplicar lógica completa

## ResolveTenantNotificationPolicy

- obtiene política de notificación efectiva
- define canales, fallback y destinatarios

## ResolveTenantAIProfile

- obtiene perfil AI activo del tenant
- aporta prompts, tolerancias y políticas de automatización

## ResolveTenantSchedule

- obtiene calendario y horario operativo vigente
- útil para decisiones contextuales

# Arquitectura recomendada

## Separar defaults de overrides

No guardes todo solo como JSON suelto.  
Conviene distinguir:

- configuración base/global
- configuración efectiva del tenant
- overrides puntuales

Eso hace mucho más fácil:

- auditar
- validar
- extender
- depurar

## Config efectiva resoluble

Muchos módulos van a necesitar una vista consolidada, así que conviene tener algo tipo:

- `ResolvedTenantConfig`
- `TenantOperationalConfigView`

para no recalcular toda la jerarquía cada vez.

## Versionado y snapshots

Muy recomendable para:

- reproducir decisiones pasadas
- auditar cambios
- hacer rollback
- saber con qué política operaba el tenant en cierto momento

## Configuración por dominio

Aunque exista `TenantSetting`, conviene conservar objetos específicos como:

- `TenantNotificationPolicy`
- `TenantAIProfile`
- `TenantScheduleProfile`

porque hacen el modelo más expresivo y evitan un “JSON blob monster”.

- horario operativo extendido
- políticas distintas fuera de horario
- supervisor nocturno como destinatario principal

# Riesgos que este módulo debe prevenir

- comportamiento rígido igual para todos los tenants
- exceso de hardcode en reglas y políticas
- configuraciones imposibles de auditar
- overrides caóticos sin precedencia clara
- settings sensibles editados por usuarios incorrectos
- decisiones históricas imposibles de reproducir por cambios de configuración
- crecimiento desordenado de parámetros
