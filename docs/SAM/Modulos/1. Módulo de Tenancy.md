Es el módulo que permite que SAM funcione como **SaaS multiempresa**, garantizando aislamiento entre clientes, configuración independiente, control de acceso por empresa y medición precisa del uso para fines operativos y de facturación.

## Responsabilidad

Administrar:

- los clientes (tenants) de la plataforma
- el aislamiento lógico de datos
- la pertenencia de usuarios a una empresa
- la configuración independiente por tenant
- los módulos/features habilitados
- el plan comercial y estado de suscripción
- la medición detallada de uso consumido por tenant y por usuario

---

## Qué resuelve

- Separación estricta por cliente
- Acceso únicamente a los datos del tenant correspondiente
- Configuración independiente por empresa
- Branding y parametrización por tenant
- Habilitación/deshabilitación de módulos por cliente
- Relación entre usuarios y empresa
- Base para planes, límites, cuotas y facturación
- Medición de usage real para cobro y auditoría

---

## Alcance del módulo

Este módulo gobierna todo lo relacionado con el **contexto empresarial** dentro de SAM:

- quién es el cliente
- qué usuarios pertenecen a ese cliente
- qué puede usar
- cuánto ha consumido
- qué límites tiene
- cómo se le cobrará

No ejecuta la lógica operativa de monitoreo en sí, pero sí define el **marco de control** bajo el cual esa lógica corre.

# Reglas de negocio

## Aislamiento

- Un usuario solo puede ver datos de su tenant
- Toda consulta de negocio debe resolverse dentro del contexto del tenant activo
- Ningún recurso operativo debe existir sin `tenant_id`
- Los jobs, eventos, cachés y reportes deben conservar contexto de tenant

## Membresía y acceso

- Un usuario solo opera dentro de los tenants a los que pertenece
- Los permisos dependen de rol + features habilitados + estado de suscripción
- Si la suscripción está suspendida, ciertos módulos pueden bloquearse

## Configuración

- Cada tenant puede tener reglas operativas propias
- Cada tenant puede definir horarios, branding, idioma y políticas
- La configuración del tenant puede sobrescribir defaults globales

## Features y planes

- Cada tenant puede habilitar o deshabilitar módulos
- El plan determina features base y límites incluidos
- Overrides manuales pueden existir por razones comerciales

## Usage y cobro

- Todo consumo facturable debe registrarse como `UsageEvent`
- Los contadores agregados no sustituyen el log granular; lo complementan
- El cobro debe derivarse de eventos auditables
- Deben existir métricas por tenant y, cuando aplique, por usuario
- Las cuotas deben poder evaluarse en tiempo real o casi real
- El sistema debe distinguir entre uso operativo y uso billable

## Nivel tenant

- incidentes generados
- reportes exportados

## Nivel feature

- uso de APIs externas
- uso de dashboards avanzados

# Estrategia de cobro sugerida

Para SAM, una buena combinación sería:

## Base fija + usage variable

### Cargo base

Por tenant:

- acceso a la plataforma
- cierto número de usuarios
- cierto número de activos
- módulos base

### Cargo variable

Por consumo:

- activos extra
- análisis AI
- almacenamiento
- notificaciones premium
- uso de módulos avanzados
- exceso de eventos procesados

Eso te da un esquema más sano que cobrar solo por usuarios.

## Resolución del tenant activo

El tenant puede resolverse por:

- membership del usuario autenticado
- subdominio
- header de API
- contexto de sesión
- custom domain en clientes enterprise

---

## Lugares donde no debes olvidar el tenant

- controladores
- queries Eloquent
- jobs en cola
- eventos de dominio
- listeners
- cache keys
- archivos exportados
- reportes PDF/Excel
- webhooks
- logs y auditoría
- métricas y usage events

# Flujos principales del módulo

## 1. Alta de tenant

1. Se crea tenant
2. Se crea owner
3. Se asigna plan
4. Se crean settings default
5. Se habilitan features iniciales
6. Se crea periodo de prueba o suscripción
7. Se inicializan contadores de usage

## 2. Acceso de usuario

1. Usuario autentica
2. Sistema determina memberships
3. Selecciona tenant activo
4. Se cargan roles, features y límites
5. Se aplica contexto global de tenant

## 3. Registro de usage

1. Ocurre una acción billable
2. Se emite evento de dominio
3. Job crea `UsageEvent`
4. Proceso agrega a `TenantUsageCounter`
5. Se evalúan límites/overages
6. Se actualiza dashboard de consumo

## 4. Cierre de facturación

1. Se toman usage events del periodo
2. Se agrupan por métrica
3. Se aplican reglas del plan
4. Se calculan incluidos y excedentes
5. Se genera snapshot facturable
6. Se emite invoice o cargo externo

---

# Riesgos que este módulo debe prevenir

- fuga de datos entre tenants
- errores de facturación por consumo mal medido
- doble conteo de usage
- pérdida de contexto tenant en colas
- features habilitados incorrectamente
- usuarios viendo datos de otra empresa
- inconsistencia entre plan, límites y consumo real
