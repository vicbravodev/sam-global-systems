Es el módulo encargado de tomar la decisión final del sistema a partir de la evaluación AI, el contexto operativo y las reglas de negocio configuradas por cada tenant.

Este módulo traduce el análisis inteligente en una **acción gobernada por políticas del negocio**, permitiendo que SAM actúe de forma consistente, auditable y personalizable según las necesidades operativas de cada cliente.

---

## Responsabilidad

Administrar:

- la decisión final del sistema sobre cada evento evaluado
- la combinación de AI, reglas del cliente y contexto operativo
- la aplicación de reglas configurables por tenant
- la resolución de rutas de escalamiento
- la ejecución o preparación de automatizaciones
- la definición de cuándo actuar automáticamente y cuándo requerir intervención humana
- la trazabilidad de por qué SAM tomó una decisión específica

---

## Qué resuelve

- decisión final del sistema
- personalización por tenant
- automatización controlada
- escalamiento según políticas
- gobernanza sobre la AI
- consistencia operativa entre eventos similares
- control de riesgo en acciones automáticas
- base auditable para incidentes, alertas y revisión humana

---

## Alcance del módulo

Este módulo gobierna la transición entre:

- lo que la AI **sugiere**
- y lo que SAM **realmente hace**

Su propósito es responder preguntas como:

- ¿debe ignorarse este evento o tratarse?
- ¿debe solo registrarse o generar alerta?
- ¿debe crearse un incidente?
- ¿debe escalarse inmediatamente?
- ¿debe ejecutarse una acción automática?
- ¿debe enviarse a revisión humana?
- ¿las reglas del tenant modifican la recomendación AI?

Este módulo no necesariamente ejecuta toda la operación completa, pero sí determina el **outcome oficial** que desencadena el resto del sistema.

---

# Inputs del módulo

Este módulo consume:

- `NormalizedEvent`

- `OperationalContextProfile`
- `AIEventEvaluation`
- reglas y políticas del tenant

- features y límites del tenant

# Posibles resultados del motor

## IGNORE

El evento no requiere acción operativa.

## LOG_ONLY

El evento se conserva para trazabilidad, pero no genera atención.

## ALERT

El evento genera una alerta visible o notificable.

## INCIDENT

El evento amerita apertura de incidente operativo.

## ESCALATE

El evento debe escalarse por política definida.

## REQUIRE_HUMAN_REVIEW

El sistema no actúa automáticamente y lo envía a validación humana.

## Contexto operativo

- geocerca
- activo

- políticas específicas por tenant

## Gobernanza y seguridad

- si la confianza AI es baja

- si la política exige revisión humana

# Ejemplos de reglas

## 1. Botón de pánico + riesgo alto

- si `event_type = panic_button`
- y `risk_score >= high`
- entonces:
    - `outcome = INCIDENT`
    - prioridad = crítica
    - aplicar escalación crítica

## 2. Cámara obstruida prolongada

- si `event_type = camera_obstructed`
- y duración > X minutos
- entonces:
    - `outcome = ESCALATE`

## 3. Colisión en geocerca sensible

- si `event_type = collision`
- y `is_in_sensitive_geofence = true`
- entonces:
    - aumentar prioridad
    - forzar escalación

## 4. Baja confianza AI

- si `confidence_score < threshold`
- entonces:
    - `outcome = REQUIRE_HUMAN_REVIEW`

## 5. Media pendiente en evento crítico

- si `event_type = collision`
- y `media_pending = true`
- entonces:
    - crear alerta inmediata
    - permitir reevaluación al llegar evidencia
    - no cerrar decisión como final hasta revisión posterior

# Reglas de negocio

## La AI no decide sola

- la evaluación AI es una entrada, no la autoridad final
- la decisión final siempre debe pasar por políticas del negocio

## Personalización por tenant

- cada tenant puede tener sus propias reglas y políticas
- el mismo evento puede resolverse distinto según cliente

## Automatización gobernada

- no todas las decisiones pueden automatizarse
- debe poder definirse qué acciones requieren confirmación humana

## Trazabilidad obligatoria

- toda decisión debe poder explicarse
- debe quedar claro qué regla o señal modificó el resultado

## Determinismo razonable

- para el mismo input y mismo ruleset, el sistema debería producir el mismo resultado
- evita comportamiento errático

## Reevaluación y actualización

- si llega nueva evidencia o cambia el contexto, una decisión puede revisarse
- las reevaluaciones deben conservar historial

# Flujos principales

## 1. Resolución de decisión

1. Llega `AIEventEvaluation`
2. Se carga `RuleSet` aplicable
3. Se evalúan `DecisionRule`
4. Se combinan señales AI + contexto + políticas
5. Se determina outcome final
6. Se crea `Decision`
7. Se genera `DecisionTrace`

---

## 2. Escalamiento

1. La decisión final requiere escalación
2. Se resuelve `EscalationPolicy`
3. Se determinan pasos
4. Se generan acciones
5. Se pasa a ejecución operativa

---

## 3. Automatización controlada

1. La decisión incluye automatización
2. Se validan restricciones del tenant
3. Se preparan `AutomationAction`
4. Se ejecutan en modo sync/async
5. Se registra resultado

---

## 4. Revisión humana

1. El motor determina incertidumbre o baja confianza
2. Outcome = `REQUIRE_HUMAN_REVIEW`
3. Se enruta a cola humana
4. Se conserva la explicación y evidencia usada

---

## 5. Reevaluación

1. Llega nueva media o cambia contexto
2. Se reevalúa AI si aplica
3. El motor vuelve a correr reglas
4. Se genera nueva decisión o actualización
5. Se conserva historial completo

---

# Servicios principales

## EvaluateDecisionRules

- evalúa reglas aplicables
- combina señales y políticas
- resuelve outcome final

## ResolveEscalationPath

- carga y aplica configuración específica del cliente
- soporta overrides y versionado

## GenerateDecisionTrace

- registra trazabilidad detallada
- explica por qué se tomó la decisión

# Jobs

## RunDecisionEngineJob

- toma evaluaciones AI pendientes

- persiste decisión y trazabilidad

## ExecuteAutomationActionJob

- ejecuta acciones automáticas derivadas de una decisión
- maneja reintentos y observabilidad

## ReevaluateDecisionJob

- vuelve a ejecutar el motor cuando cambia evidencia o contexto
- versiona decisiones posteriores

# Arquitectura recomendada

## Separar evaluación de decisión

Esto es importantísimo:

- AI evalúa
- Decision Engine decide
- Workflows ejecutan

No mezclar esas capas te va a ahorrar mucho dolor después.

## Rule engine configurable

- evitar hardcode excesivo
- permitir reglas por tenant
- soportar versionado y prioridad
- permitir stop-processing cuando una regla crítica aplica

## Fusión jerárquica

Yo usaría una lógica así:

1. hard safety rules
2. tenant overrides
3. AI recommendation
4. escalation policies
5. fallback default outcome

Así mantienes control del negocio por encima de la AI.

## Outcomes limitados y claros

No conviene tener outcomes infinitos.  
Mejor un set pequeño, claro y estable, con acciones derivadas configurables.

# Riesgos que este módulo debe prevenir

- dejar que la AI actúe sin gobernanza
- decisiones inconsistentes entre tenants
- automatizaciones peligrosas sin control
- reglas opacas o difíciles de auditar
- outcomes ambiguos
- escalamiento incorrecto
- exceso de hardcode no configurable
- no reaccionar ante nueva evidencia
