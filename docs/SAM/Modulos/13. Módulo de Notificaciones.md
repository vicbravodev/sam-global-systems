Es el módulo encargado de gestionar el envío de avisos y comunicaciones operativas a los canales adecuados según el tenant, la prioridad, el tipo de evento, incidente o acción ejecutada.

Este módulo abstrae la lógica de entrega de mensajes para que SAM pueda comunicar alertas, escalaciones y actualizaciones de forma consistente, configurable y auditable, sin mezclar la mensajería con el motor principal del negocio.

---

## Responsabilidad

Administrar:

- el envío de notificaciones a múltiples canales
- la resolución de destinatarios según contexto y políticas
- la selección y renderizado de plantillas
- la trazabilidad de intentos de entrega
- el manejo de fallos y reintentos
- la personalización por tenant, prioridad y tipo de evento
- la separación entre intención de notificar y entrega efectiva

---

## Qué resuelve

- envío de avisos por distintos canales
- selección del canal adecuado según criticidad o política
- personalización de mensajes por tenant
- resolución dinámica de destinatarios
- reintentos ante fallas de entrega
- trazabilidad completa de la notificación
- desacoplamiento entre negocio y comunicación
- soporte multi-canal y multi-destinatario

---

## Alcance del módulo

Este módulo gobierna todo lo relacionado con la **comunicación saliente del sistema**:

- qué mensaje se enviará
- por qué canal
- a quién
- con qué plantilla
- con qué prioridad
- si fue entregado, falló o requiere reintento

Su propósito es responder preguntas como:

- ¿a quién hay que avisar?
- ¿por qué canal conviene notificar?
- ¿qué contenido debe enviarse?
- ¿la notificación fue entregada o falló?
- ¿hay que reintentar o cambiar de canal?
- ¿el tenant permite WhatsApp, Slack o SMS?

Este módulo no decide cuándo existe un incidente o una escalación, sino que se encarga de **comunicar correctamente** lo que otros módulos determinaron.

# Reglas de negocio

## Separación del negocio

- este módulo no decide qué hacer operativamente
- solo se encarga de comunicar de forma correcta y trazable

## Multi-canal

- una notificación puede entregarse por varios canales
- el canal puede depender de prioridad, tipo y políticas del tenant

## Destinatarios dinámicos

- los destinatarios pueden resolverse por:
    - usuario
    - rol
    - equipo
    - cola
    - contacto externo
- no todo debe estar hardcodeado

## Templates y personalización

- el contenido debe renderizarse con plantillas
- los tenants pueden personalizar mensajes, branding e idioma

## Retries y fallback

- si un canal falla, puede reintentarse
- también puede definirse fallback a otro canal
- no todos los canales deben tener la misma política de retry

## Auditoría

- toda notificación debe quedar registrada
- debe saberse si fue enviada, entregada, falló o reintentó

## Priorización

- las notificaciones críticas deben poder saltarse canales lentos o secundarios
- el sistema debe soportar urgencia y SLA de comunicación

# Flujos principales

## 1. Creación de notificación

1. Otro módulo solicita notificar
2. Se crea `Notification`
3. Se resuelve tipo y prioridad
4. Se selecciona plantilla
5. Se resuelven destinatarios
6. Se generan deliveries

---

## 2. Resolución de destinatarios

1. Se analiza el contexto del incidente/evento
2. Se consultan reglas, roles y preferencias
3. Se generan `NotificationRecipient`
4. Se decide canal preferido o permitido

---

## 3. Renderizado de plantilla

1. Se selecciona `NotificationTemplate`
2. Se inyectan variables del contexto
3. Se genera contenido final
4. Se prepara payload por canal

---

## 4. Envío

1. Se toma `NotificationDelivery` pendiente
2. Se selecciona proveedor/canal
3. Se ejecuta envío
4. Se registra respuesta
5. Se marca como delivered o failed

---

## 5. Retry o fallback

1. Falla una entrega
2. Se evalúa política de retry
3. Se reintenta o se cambia de canal
4. Se registra nuevo `NotificationDelivery`

---

# Servicios principales

## SendNotification

- crea y despacha una notificación
- coordina plantilla, destinatarios y canales

## ResolveRecipients

- resuelve a quién se debe notificar
- usa reglas, roles, contexto y preferencias

## RenderTemplate

- genera el contenido final del mensaje
- soporta variables dinámicas y plantillas por canal

## RetryFailedNotification

- reintenta deliveries fallidos
- aplica política de backoff o fallback

## SelectNotificationChannels

- determina canales apropiados según prioridad y tenant

## DispatchNotificationDelivery

- ejecuta la entrega técnica en el canal correspondiente

# Arquitectura recomendada

## Separar intención de entrega

Muy importante distinguir entre:

- `Notification` = intención de comunicar
- `NotificationDelivery` = intento técnico de entrega

Eso te da trazabilidad real y mucha flexibilidad.

## Multi-canal first

Diseña el módulo para que una misma notificación pueda:

- ir a varios destinatarios
- por varios canales
- con varios resultados de entrega

## Plantillas por canal

No uses un solo texto para todo.  
Cada canal debe poder tener su propia plantilla:

- email largo
- SMS corto
- WhatsApp estructurado
- Slack resumido

## Fallback y observabilidad

Las fallas de comunicación no deben ser silenciosas.  
Debe poder saberse:

- qué falló
- por qué
- si hubo retry
- si hubo fallback

# Ejemplos de reglas útiles

## Crítico

- si prioridad = critical
- canales: WhatsApp + SMS + Slack
- destinatarios: supervisor + monitorista + contacto crítico

## Medio

- si prioridad = medium
- canal: email + web notification

## Revisión humana

- si outcome = `REQUIRE_HUMAN_REVIEW`
- canal: web notification + Slack interno

## Tenant enterprise

- además de notificar personas
- enviar webhook saliente al sistema del cliente

# Riesgos que este módulo debe prevenir

- notificaciones perdidas
- envíos duplicados
- mensajes enviados al destinatario equivocado
- contenido inconsistente entre canales
- fallos silenciosos de entrega
- hardcode excesivo de canales y mensajes
- mezclar lógica de negocio con lógica de comunicación
