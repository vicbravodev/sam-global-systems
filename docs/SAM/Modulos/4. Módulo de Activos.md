Es el módulo encargado de representar y gestionar los activos monitoreados dentro de SAM, tales como vehículos, cámaras, dispositivos GPS, sensores u otros elementos que generan datos operativos.

Este módulo actúa como la **fuente de verdad interna operativa** sobre los activos dentro de SAM, pero su creación y actualización base dependen de los proveedores externos integrados, garantizando sincronización continua con la realidad operativa del cliente.

---

## Responsabilidad

Administrar:

- la representación interna de los activos sincronizados desde proveedores externos
- la clasificación por tipo de activo
- la relación entre activos y dispositivos
- la asociación con identificadores externos
- el estado actual de cada activo
- snapshots de ubicación y contexto geográfico
- la consolidación de datos provenientes de integraciones

---

## Qué resuelve

- representación unificada de activos físicos
- desacoplamiento entre proveedores externos y modelo interno
- sincronización consistente del inventario operativo
- estado actual confiable de cada activo
- vinculación entre dispositivos y activos
- base estructural para eventos, alertas y monitoreo
- normalización de datos de múltiples fuentes

---

## Alcance del módulo

Este módulo gobierna todo lo relacionado con la **identidad, estado y contexto operativo de los activos monitoreados**:

- qué activos existen en el sistema
- qué tipo de activos son
- qué dispositivos tienen asociados
- cuál es su estado actual
- dónde se encuentran
- cómo se relacionan con proveedores externos

No permite el alta manual de activos. Los activos existen en SAM únicamente cuando han sido descubiertos o sincronizados desde un proveedor externo autorizado.

No procesa eventos ni toma decisiones por sí mismo, pero provee el **contexto estructural** sobre el cual operan el engine, las alertas y los análisis.

# Entidades principales

## 1. Asset

Representa un activo físico monitoreado.

### Campos sugeridos

- id
- tenant_id
- asset_type_id
- provider_id
- source_integration_id
- external_primary_id
- name
- code / internal_identifier
- status
- metadata_json
- first_seen_at
- last_seen_at
- created_at
- updated_at

### Ejemplos

- unidad 1023
- tráiler 55-A
- cámara frontal cam-22
- sensor temperatura caja

### Notas

- Es la entidad central del módulo
- Todo asset debe originarse desde una integración externa
- Todo evento debe poder vincularse a un `asset_id`
- SAM no crea activos manualmente; los materializa a partir del provider

## 3. AssetStatus

Estado actual del activo derivado de eventos recientes.

### Campos sugeridos

- id
- asset_id

- updated_at

## 5. AssetLocationSnapshot

Última ubicación conocida o snapshots históricos del activo.

### Campos sugeridos

- id
- asset_id
- latitude
- longitude
- formatted_location
- speed
- heading
- recorded_at
- source
- geocoding_metadata_json

### `formatted_location`

Campo de dirección legible para humanos, por ejemplo:

- Av. Constitución 1234, Monterrey, Nuevo León, México
- Km 45 Carretera Saltillo-Monterrey, Santa Catarina, Nuevo León
- Calle Hidalgo 210, Centro, San Luis Potosí, México

### Notas

- Además de latitud/longitud, SAM debe almacenar una versión human-readable de la ubicación
- Esto mejora monitoreo, auditoría, reportes y atención operativa
- Puede provenir del provider o derivarse por geocodificación interna/externa
- Es especialmente útil para que el monitorista o agente AI entienda rápido dónde ocurrió algo

# Reglas de negocio

## Origen de activos

- todo activo debe pertenecer a un tenant
- ningún asset puede existir sin `tenant_id`
- los activos no pueden crearse manualmente desde la UI o por operación interna
- todo asset debe originarse desde una integración válida
- el provider externo es la fuente maestra del inventario del activo

## Relación con integraciones

- un activo puede tener múltiples referencias externas
- un asset puede estar vinculado a múltiples dispositivos
- los datos externos deben mapearse a un asset interno
- si el activo deja de existir en el provider, SAM debe decidir si marcarlo como inactive, archived o detached, pero no eliminarlo físicamente sin política explícita

## Estado

- el estado del activo debe derivarse de eventos recientes
- debe existir un único estado actual por activo
- cambios de estado deben ser trazables

## Ubicación

- la ubicación debe actualizarse con eventos de telemetría
- debe existir al menos un último snapshot válido
- la ubicación debe almacenar coordenadas y dirección legible
- `formatted_location` debe representar una dirección comprensible para humanos

## Consistencia

- los assets no deben duplicarse por proveedor
- el mapping externo debe ser único por provider + external_id
- SAM no debe depender únicamente del nombre del asset para identificarlo

# Flujos principales del módulo

## 1. Descubrimiento inicial desde integración

1. Se conecta un provider al tenant
2. Se ejecuta sincronización inicial
3. SAM obtiene inventario de activos del proveedor
4. Se crean registros `Asset`
5. Se crean referencias externas
6. Se inicializa estado y contexto base

---

## 2. Sincronización de activos desde integración

1. Llega data externa o corre sync programado
2. Se busca mapping existente
3. Si no existe, se crea asset automáticamente
4. Si existe, se actualiza asset
5. Se actualizan referencias y dispositivos
6. Se actualiza `last_seen_at`

---

## 3. Actualización de estado

1. Llega evento
2. Se identifica asset
3. Se evalúa impacto operativo
4. Se actualiza `AssetStatus`
5. Se registra fuente del cambio

---

## 4. Actualización de ubicación

1. Llega evento de ubicación
2. Se identifica asset
3. Se guardan coordenadas
4. Se resuelve `formatted_location`
5. Se actualiza snapshot más reciente

---

# Servicios principales

## SyncAssetFromIntegration

- crea o actualiza asset desde proveedor externo
- maneja mapping externo
- evita duplicados
- sincroniza atributos base del inventario

## UpdateAssetStatus

- actualiza estado basado en eventos
- aplica reglas de prioridad
- mantiene trazabilidad

## AttachDeviceToAsset

- vincula dispositivo a asset
- registra relación histórica
- conserva contexto del provider

## UpdateAssetLocationSnapshot

- almacena coordenadas recientes
- resuelve dirección legible
- actualiza última ubicación conocida

# Arquitectura recomendada

## Asset como fuente de verdad interna

- el provider externo es la fuente de creación/origen
- SAM es la fuente de verdad operativa interna una vez normalizado
- todo evento debe resolverse a un `asset_id`
- el engine nunca debe trabajar con IDs externos directamente

## Sin creación manual

- no debe existir `RegisterAsset` manual
- los activos deben nacer por sincronización
- cualquier alta de activo fuera de integración introduce riesgo de desalineación con el provider

## Normalización obligatoria

- ningún módulo debe depender del formato del proveedor
- todo pasa por `Asset` como entidad central

---

# Riesgos que este módulo debe prevenir

- duplicación de activos
- desalineación entre provider y SAM
- pérdida de trazabilidad
- assets sin mapping claro
- estados incorrectos o desactualizados
- ubicaciones sin contexto legible para operación
- activos “fantasma” creados manualmente fuera del provider
