Es el módulo encargado de analizar eventos enriquecidos con contexto operativo y evidencia disponible para determinar su significado real, nivel de riesgo, prioridad y acciones sugeridas, utilizando modelos de inteligencia artificial, lógica heurística y evaluación multimodal.

Este módulo simula el criterio de un monitorista humano, evaluando cada evento no solo por su tipo y contexto, sino también por la **evidencia multimedia asociada**, permitiendo distinguir mejor entre eventos reales, falsos positivos, ruido operativo o situaciones críticas que requieren acción inmediata.

---

## Responsabilidad

Administrar:

- el análisis inteligente de eventos normalizados con contexto operativo
- la evaluación multimodal usando telemetría, contexto y media
- la detección de falsos positivos
- la priorización de eventos
- la determinación de riesgo y criticidad real
- la generación de decisiones sugeridas o automáticas
- la combinación de reglas, scoring y modelos AI
- la explicabilidad de decisiones
- la reevaluación de eventos cuando llega nueva evidencia multimedia

---

## Qué resuelve

- distinguir entre eventos reales y ruido
- reducir falsas alarmas
- priorizar eventos críticos
- identificar patrones de riesgo
- validar eventos con evidencia visual o multimedia
- detectar inconsistencias entre telemetría y media
- sugerir o ejecutar acciones operativas
- reducir carga operativa humana
- mejorar tiempos de respuesta
- aumentar confianza del usuario en las decisiones del sistema

---

## Alcance del módulo

Este módulo gobierna la **interpretación inteligente del evento** una vez que SAM ya cuenta con:

- evento normalizado
- contexto operativo
- evidencia multimedia disponible o estado de disponibilidad de media

Su propósito es responder preguntas como:

- ¿es un evento real o un falso positivo?
- ¿qué tan grave es realmente?
- ¿requiere acción inmediata?
- ¿la evidencia disponible confirma o contradice el evento?
- ¿la ausencia de media reduce la confianza?
- ¿conviene esperar evidencia adicional antes de escalar?
- ¿qué acción operativa debería tomarse?

Este módulo no necesariamente ejecuta directamente todas las acciones finales, pero sí produce la **evaluación principal** que el sistema utilizará para decidir cómo actuar.

# Qué analiza este módulo

Para SAM, la evaluación inteligente ideal combina:

## Señales del evento

- tipo de evento
- categoría
- severidad normalizada
- source/provider
- timestamp

## Contexto operativo

- ubicación
- geocerca
- estado del activo
- historial reciente
- incidentes abiertos
- reincidencia
- conductor asignado
- señales operativas derivadas

## Evidencia multimedia

- existencia de video o imagen
- evidencia previa, durante o posterior al evento
- confirmación visual
- calidad de la media
- cámara disponible/no disponible
- media pendiente o expirada
- contradicción entre media y telemetría

## Condiciones de negocio

- criticidad del tenant
- tipo de operación
- políticas del cliente
- reglas específicas de escalación
- SLA objetivo

# Reglas de negocio

## Evaluación obligatoria

- todo evento relevante debe pasar por evaluación AI o heurística
- no debe haber decisiones operativas sin evaluación previa, salvo reglas críticas excepcionales

## Multimodalidad gradual

- la evaluación puede iniciar sin media
- si la media llega después, el sistema debe poder reevaluar
- la ausencia de media no debe bloquear el pipeline

## Explicabilidad

- toda decisión debe poder explicarse
- el sistema debe guardar el razonamiento y las señales que influyeron
- debe quedar claro si la decisión se tomó con evidencia o sin ella

## Consistencia

- eventos similares deben producir evaluaciones similares
- la presencia de media debe aumentar precisión, no volver errático el comportamiento

## Híbrido AI + reglas

- no todo debe depender de un LLM
- combinar:
    - reglas determinísticas
    - scoring heurístico
    - evaluación AI textual
    - evaluación multimodal cuando aplique
- debe existir fallback si un componente falla

## Reevaluación controlada

- un evento puede reevaluarse si llega nueva evidencia
- las reevaluaciones deben quedar versionadas
- no debe perderse la decisión original ni el contexto usado en ese momento

# Flujos principales

## 1. Evaluación inicial sin media

1. Llega evento con contexto
2. Se construye input estructurado
3. Se ejecuta pipeline híbrido
4. Se calcula riesgo, prioridad y clasificación
5. Si la media está pendiente, se puede marcar como `pending_evidence`
6. Se guarda `AIEventEvaluation`

---

## 2. Evaluación multimodal con media disponible

1. Llega evento con contexto y evidencia
2. Se analiza la media asociada
3. Se generan señales de validación visual o auditiva
4. Se combinan con telemetría y contexto
5. Se produce evaluación final
6. Se guardan `AIEventEvaluation` y `AIMediaAssessment`

---

## 3. Detección de falso positivo

1. Evento llega
2. Contexto indica baja relevancia o contradicción
3. La media no confirma el evento o lo contradice
4. Se marca como `false_positive`
5. Se evita escalación innecesaria

---

## 4. Reevaluación por nueva evidencia

1. El evento ya tenía evaluación inicial
2. Llega clip, snapshot o nueva media
3. Se crea `AIReevaluationRequest`
4. Se vuelve a ejecutar análisis
5. Se crea nueva versión de `AIEventEvaluation`
6. Si cambia el resultado, se actualiza el flujo operativo

---

## 5. Generación de acciones

1. Se completa evaluación
2. La AI determina recomendación
3. Se generan acciones sugeridas
4. Se pasan al módulo de ejecución o incidentes

# Pipeline recomendado

## Etapa 1: filtros rápidos

- duplicados
- reglas determinísticas críticas
- validaciones técnicas
- priorización mínima

## Etapa 2: scoring heurístico

- riesgo base
- reincidencia
- geocerca sensible
- incidentes abiertos
- estado del activo y conductor

## Etapa 3: AI textual/contextual

- interpretación del evento y contexto
- clasificación preliminar
- explicación

## Etapa 4: evaluación multimodal

- análisis de imagen, clip o audio
- confirmación o contradicción del evento

## Etapa 5: fusión de decisión

- combinar señales de todas las etapas
- producir resultado final
- sugerir acciones

# Arquitectura recomendada

## Evaluación híbrida y multimodal

- no dependas de una sola llamada a AI
- separa:
    - razonamiento contextual
    - scoring
    - evaluación de evidencia
- luego fusiona el resultado

## Versionado fuerte

- cada evaluación debe tener versión
- cada reevaluación debe conservar trazabilidad
- útil para auditoría, debugging y mejora continua

## Control de costo

- no mandes media a AI si no aporta valor
- define tipos de evento donde la media es:
    - obligatoria
    - útil
    - opcional
- controla tokens, duración de clips y frecuencia de reevaluación

## Explicabilidad first-class

- siempre guardar:
    - qué señales influyeron
    - qué evidencia se usó
    - qué faltaba
    - por qué se sugirió cierta acción

# Riesgos que este módulo debe prevenir

- falsas alarmas
- saturación operativa
- decisiones inconsistentes
- dependencia total de AI sin fallback
- falta de explicabilidad
- alto costo de inferencia
- ignorar evidencia relevante
- tomar decisiones fuertes sin media cuando debió esperarse
- no reevaluar cuando llegó evidencia crítica
