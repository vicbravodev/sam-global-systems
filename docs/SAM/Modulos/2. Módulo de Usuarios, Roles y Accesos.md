Es el módulo encargado de gestionar la identidad de los usuarios dentro de SAM, garantizando que cada persona tenga acceso únicamente a los recursos y acciones permitidas según su rol, permisos y contexto de tenant.

Este módulo controla la autenticación, autorización, membresía por empresa y la trazabilidad básica de las acciones realizadas por los usuarios dentro del sistema.

---

## Responsabilidad

Administrar:

- la identidad de los usuarios de la plataforma
- los mecanismos de autenticación y sesiones
- la pertenencia de usuarios a uno o múltiples tenants
- los roles y permisos dentro de cada organización
- las reglas de autorización para cada acción del sistema
- las preferencias individuales de usuario
- el registro básico de actividad para auditoría y seguridad

---

## Qué resuelve

- control de acceso seguro al sistema
- autenticación de usuarios (login, sesiones, etc.)
- autorización basada en roles y permisos
- asignación de usuarios a tenants específicos
- diferenciación de capacidades entre tipos de usuario
- segregación de funciones dentro de una organización
- trazabilidad de acciones críticas
- base para seguridad operativa y cumplimiento

---

## Alcance del módulo

Este módulo gobierna todo lo relacionado con la **identidad y control de acceso** dentro de SAM:

- quién es el usuario
- a qué empresa(s) pertenece
- qué rol tiene dentro de cada tenant
- qué acciones puede ejecutar
- bajo qué condiciones puede acceder a recursos

No define qué features están disponibles para un tenant (eso pertenece al Módulo 1), pero sí determina **quién puede usarlas y cómo**.

# Entidades principales

## 1. User

Representa a una persona que puede acceder al sistema.

### Campos sugeridos

- id
- name
- email
- password
- status
- last_login_at
- email_verified_at
- global_role (opcional)
- created_at
- updated_at

### Notas

- Es una identidad global dentro del sistema
- Puede pertenecer a uno o múltiples tenants
- No define directamente su rol operativo (eso vive en la membresía)

---

## 2. TeamMembership

Representa la relación entre un usuario y un tenant, incluyendo su rol dentro de esa organización.

### Campos sugeridos

- id
- tenant_id
- user_id
- role
- status
- invited_by
- joined_at
- last_active_at

### Notas

- Es la entidad puente entre este módulo y el de Tenancy
- Define el **rol efectivo del usuario dentro de un tenant**
- Permite que un mismo usuario tenga distintos roles en diferentes empresas

---

## 3. Role

Define un conjunto de permisos agrupados.

### Campos sugeridos

- id
- name
- code
- description
- scope (global | tenant)
- is_system
- created_at

### Uso

- Agrupa permisos para simplificar la asignación
- Puede existir tanto a nivel global como por tenant

---

## 4. Permission

Define una capacidad específica dentro del sistema.

### Campos sugeridos

- id
- code
- name
- description
- module
- created_at

### Ejemplos

- incidents.view
- incidents.manage
- alerts.resolve
- users.manage
- reports.export
- ai.analysis.execute

---

## 5. UserPreference

Configuraciones individuales del usuario.

### Campos sugeridos

- id
- user_id
- tenant_id nullable
- preferences_json

### Ejemplos

- idioma
- tema UI
- configuración de notificaciones
- dashboards favoritos

---

## 6. UserSession

Gestión de sesiones activas.

### Campos sugeridos

- id
- user_id
- ip_address
- user_agent
- last_activity_at
- expires_at

---

## 7. ActivityLog / AuditLog

Registro de acciones relevantes realizadas por usuarios.

### Campos sugeridos

- id
- user_id
- tenant_id
- action
- entity_type
- entity_id
- metadata_json
- created_at

### Ejemplos

- login exitoso
- cambio de rol
- creación de usuario
- cierre de incidente
- exportación de reporte

# Reglas de negocio

## Autenticación

- Todo acceso al sistema requiere autenticación válida
- Las sesiones deben ser seguras y expirables
- Se debe poder invalidar sesiones activas

## Autorización

- Toda acción debe validarse contra permisos explícitos
- Los permisos se evalúan en el contexto del tenant activo
- El acceso depende de rol + permisos + estado del tenant

## Membresía

- Un usuario solo puede operar dentro de los tenants a los que pertenece
- Un usuario puede tener distintos roles en distintos tenants
- La membresía puede estar activa, suspendida o pendiente

## Seguridad

- Ningún usuario puede acceder a datos fuera de su ámbito autorizado
- Acciones sensibles deben quedar registradas
- Cambios de roles y permisos deben auditarse

## Consistencia con Tenancy

- El acceso debe respetar features habilitados por tenant
- Si un tenant está suspendido, el acceso puede limitarse
- El contexto de tenant debe resolverse antes de cualquier validación

# Relación con otros módulos

## Con Tenancy / Multiempresa

- utiliza el tenant activo para resolver autorización
- depende de `TeamMembership` para definir roles
- respeta límites, features y estado de suscripción

## Con Monitoreo Operativo

- controla quién puede ver, gestionar o escalar incidentes
- define quién puede interactuar con alertas y eventos

## Con Auditoría y Compliance

- provee los eventos base para trazabilidad
- permite reconstruir acciones de usuarios

# Flujos principales del módulo

## 1. Registro / invitación de usuario

1. Se crea o invita usuario
2. Se asigna a un tenant
3. Se define rol
4. Se activa membresía
5. Usuario accede al sistema

---

## 2. Inicio de sesión

1. Usuario autentica
2. Se valida estado
3. Se obtienen memberships
4. Se selecciona tenant activo
5. Se carga contexto de permisos

---

## 3. Evaluación de permisos

1. Usuario intenta acción
2. Se identifica tenant activo
3. Se obtiene rol desde membership
4. Se validan permisos asociados
5. Se permite o rechaza la acción

---

## 4. Auditoría de actividad

1. Usuario ejecuta acción
2. Se emite evento
3. Se registra en ActivityLog
4. Se almacena contexto (tenant, entidad, metadata)

---

# Riesgos que este módulo debe prevenir

- accesos no autorizados
- escalamiento indebido de privilegios
- fuga de información entre tenants
- usuarios con permisos incorrectos
- falta de trazabilidad en acciones críticas
- inconsistencias entre rol y permisos efectivos
