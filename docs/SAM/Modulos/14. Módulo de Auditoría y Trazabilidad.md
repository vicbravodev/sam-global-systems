Es el módulo encargado de registrar y preservar de forma estructurada los eventos, decisiones, cambios y acciones relevantes que ocurren dentro de SAM, permitiendo reconstruir con precisión qué pasó, cuándo pasó, quién lo hizo y por qué ocurrió.

Este módulo proporciona la **memoria auditada del sistema**, esencial para cumplimiento, soporte, análisis forense, debugging, confianza operativa y explicabilidad ante clientes enterprise.

---

## Responsabilidad

Administrar:

- el registro de acciones humanas relevantes
- la trazabilidad de eventos del sistema
- el histórico de cambios en entidades críticas
- la preservación de decisiones automáticas y sus causas
- la relación entre acciones, incidentes, reglas, notificaciones y evaluaciones AI
- la reconstrucción cronológica de un caso o flujo operativo
- la observabilidad auditada del comportamiento interno de SAM

---

## Qué resuelve

- saber quién hizo qué
- saber qué webhook entró
- saber qué decidió la IA
- saber qué regla aplicó
- saber qué incidente se creó
- saber qué notificación se envió
- reconstruir flujos operativos complejos
- justificar decisiones del sistema frente a clientes o auditorías
- facilitar debugging, soporte y análisis forense

---

## Alcance del módulo

Este módulo gobierna todo lo relacionado con el **registro explicable y trazable del sistema**.

Su propósito es responder preguntas como:

- ¿quién ejecutó esta acción?
- ¿qué evento originó este incidente?
- ¿qué evaluación AI llevó a esta decisión?
- ¿qué regla cambió el outcome?
- ¿qué cambió entre una versión y otra?
- ¿qué notificaciones se enviaron y a quién?
- ¿qué acciones automáticas se dispararon?
- ¿cómo se relaciona todo esto dentro de un mismo caso?

Este módulo no toma decisiones operativas, pero sí conserva la **evidencia estructurada** de que esas decisiones, cambios y acciones ocurrieron.

# Capas de trazabilidad recomendadas

Para SAM, conviene pensar la trazabilidad en 4 capas:

## 1. Capa funcional

Qué pasó en términos de negocio:

- se creó un incidente
- se escaló un caso
- se notificó a un supervisor

Aquí vive `AuditLog`.

## 2. Capa de dominio

Qué eventos lógicos ocurrieron:

- `ai_evaluation_completed`
- `decision_finalized`
- `incident_opened`

Aquí vive `DomainEventLog`.

## 3. Capa técnica

Cómo fluyó internamente el sistema:

- webhook recibido
- job iniciado
- retry ejecutado
- llamada externa fallida

Aquí vive `SystemTrace`.

## 4. Capa de cambios

Qué cambió exactamente:

- prioridad de high a critical
- estado de open a escalated
- regla modificada

Aquí vive `ChangeHistory`.

# Reglas de negocio

## Auditoría obligatoria de acciones críticas

- toda acción humana sensible debe auditarse
- toda decisión automática importante debe registrarse
- toda ejecución externa debe dejar rastro

## Trazabilidad de extremo a extremo

- debe poder correlacionarse un flujo completo desde evento hasta outcome final
- ningún módulo crítico debe quedar fuera de trazabilidad

## Inmutabilidad razonable

- los logs de auditoría no deberían editarse libremente
- si hay correcciones, deben quedar como nuevos registros, no sobrescribir el histórico

## Separación entre auditoría y operación

- la auditoría no sustituye los datos operativos
- es una capa complementaria, no el source of truth funcional

## Sensibilidad y privacidad

- no todo payload debe guardarse sin control
- debe poder enmascararse o excluirse información sensible
- deben existir políticas de retención y acceso

## Multi-tenant seguro

- toda auditoría debe respetar el contexto del tenant
- un tenant no debe acceder a trazas de otro tenant
- puede haber trazabilidad global solo para super admins internos

# Flujos principales

## 1. Registro de acción humana

1. Usuario ejecuta acción sensible
2. Se registra `AuditLog`
3. Si hubo cambios de entidad, se registra `ChangeHistory`
4. Se relaciona con trace o incidente si aplica

---

## 2. Registro de evento de dominio

1. Ocurre un evento relevante del sistema
2. Se emite y persiste `DomainEventLog`
3. Se asigna `correlation_id`
4. Se conecta con otros eventos del mismo flujo

---

## 3. Construcción de traza operativa

1. Entra un webhook o evento inicial
2. Se genera `trace_id`
3. Cada módulo agrega spans o registros
4. Al final, puede reconstruirse todo el recorrido

---

## 4. Auditoría de cambios

1. Una entidad crítica cambia
2. Se capturan estados before/after
3. Se registra `ChangeHistory`
4. Se asocia con actor y razón del cambio

---

## 5. Reconstrucción de caso

1. Se toma un incidente
2. Se buscan logs, eventos, traces y cambios relacionados
3. Se arma línea de tiempo unificada
4. Se explica qué pasó y por qué

# Servicios principales

## RecordAuditEvent

- registra una acción auditable
- soporta actores humanos y automáticos
- agrega resumen y contexto

## StoreDomainEvent

- persiste un evento de dominio
- mantiene correlación y causalidad

## BuildTraceForIncident

- reconstruye el flujo relacionado con un incidente
- une evento, AI, decisión, incidente, acción y notificación

## RecordEntityChange

- registra cambios estructurados before/after
- captura campos modificados y razón

## StartSystemTrace

- inicia una traza técnica
- genera trace_id y contexto de correlación

## AppendTraceSpan

- agrega pasos intermedios a una traza
- útil para jobs, integraciones y fallos

# Qué debería poder reconstruirse

Con este módulo bien hecho, SAM debería poder responder algo así:

> El 11 de abril a las 14:03 entró un webhook de Samsara.  
> Se persistió como RawEvent.  
> Luego se normalizó como `panic_button`.  
> Se construyó contexto con geocerca sensible y conductor asignado.  
> La AI lo evaluó como `true_emergency` con confianza 0.91.  
> El motor aplicó la regla `panic_high_risk_create_critical_incident`.  
> Se creó el incidente `INC-1042`.  
> Se notificó por WhatsApp y Slack al supervisor.  
> El supervisor reclasificó el incidente a `Escalated`.  
> Finalmente fue resuelto a las 14:21.

Ese nivel de narrativa vale oro en enterprise.

# Arquitectura recomendada

## Correlation ID everywhere

Este módulo se vuelve mucho más poderoso si desde el inicio manejas:

- `trace_id`
- `correlation_id`
- `causation_id`

Con eso puedes seguir flujos completos entre módulos.

## Registro asíncrono pero confiable

- mucha auditoría puede ir por cola
- pero acciones críticas deberían asegurarse de persistirse
- hay que balancear performance y confiabilidad

## Before/after selectivo

No hagas before/after gigantes para todo.  
Mejor:

- entidades críticas sí
- payloads muy pesados no siempre
- campos sensibles enmascarados

## Vista unificada para casos

Conviene luego tener una función o proyección tipo:

- `IncidentTraceView`
- `EventTraceView`

Para consultar fácil la historia completa sin cruzar mil tablas manualmente.

# Riesgos que este módulo debe prevenir

- no poder explicar por qué pasó algo
- no saber qué módulo falló
- cambios invisibles en reglas o incidentes
- falta de evidencia para auditorías de cliente
- dificultad para debugging y soporte
- pérdida de causalidad entre módulos
- mezcla desordenada entre logs técnicos y negocio
