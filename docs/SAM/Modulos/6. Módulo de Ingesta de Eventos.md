Es el módulo encargado de recibir eventos externos desde proveedores, APIs o webhooks y almacenarlos de forma segura, íntegra y auditable antes de cualquier transformación o procesamiento posterior.

Este módulo representa la **puerta de entrada confiable** de SAM para todos los eventos operativos del sistema, garantizando persistencia temprana, trazabilidad total, idempotencia y protección contra pérdida o duplicación de datos.

---

## Responsabilidad

Administrar:

- la recepción de eventos externos desde múltiples canales
- la persistencia del evento crudo tal como fue recibido
- el registro de contexto de recepción
- la trazabilidad completa del evento desde su entrada
- la detección de duplicados
- la validación inicial de integridad y origen
- la confirmación técnica de recepción hacia el proveedor cuando aplique

---

## Qué resuelve

- entrada segura de eventos por webhook
- entrada de eventos por polling o consumo de APIs externas
- almacenamiento del payload original sin alteraciones
- trazabilidad completa del evento desde el primer contacto
- detección de eventos duplicados
- reducción del riesgo de pérdida de eventos críticos
- base auditable para reprocesamiento y debugging
- desacoplamiento entre recepción y procesamiento

---

## Alcance del módulo

Este módulo gobierna todo lo relacionado con la **captura inicial y resguardo del evento externo**:

- cómo entra el evento
- desde qué proveedor o canal llega
- qué payload original se recibió
- cuándo se recibió
- si ya había sido recibido antes
- bajo qué contexto técnico llegó

Este módulo **no interpreta ni transforma el evento**. Su objetivo es asegurar que el evento exista en SAM de forma confiable antes de que otros módulos lo procesen.

# Reglas críticas de negocio

## Persistencia primero

- ningún evento debe transformarse antes de ser persistido
- el payload crudo debe guardarse completo antes de cualquier procesamiento
- si el procesamiento falla, el evento debe seguir disponible en estado recuperable

## No pérdida

- ningún evento válido debe perderse por fallo downstream
- la recepción debe estar desacoplada del procesamiento
- fallos en colas, normalización o reglas no deben afectar la persistencia inicial

## Idempotencia

- el sistema debe detectar duplicados
- un mismo evento no debe procesarse dos veces si representa la misma ocurrencia real
- la estrategia de deduplicación debe ser auditable y configurable por proveedor

## Auditoría

- todo evento recibido debe quedar registrado
- debe poder reconstruirse cuándo llegó, desde dónde y bajo qué contexto
- toda respuesta entregada al proveedor debe ser trazable

## Seguridad

- webhooks deben validar firma, secreto u otro mecanismo aplicable
- eventos inválidos o sospechosos deben registrarse con estado apropiado
- la recepción no debe exponer credenciales ni datos sensibles en logs inseguros

# Flujos principales del módulo

## 1. Recepción de webhook

1. Proveedor envía evento
2. Se identifica fuente y contexto básico
3. Se valida firma o autenticidad si aplica
4. Se persiste `RawEvent`
5. Se registra `EventReceipt`
6. Se calcula deduplication key
7. Se evalúa duplicado
8. Se responde al proveedor
9. Se encola para procesamiento posterior si corresponde

---

## 2. Recepción por polling / API externa

1. Job consulta proveedor
2. Obtiene lote de eventos
3. Por cada evento:
    - se crea `RawEvent`
    - se registra fuente
    - se calcula deduplication key
    - se detectan duplicados
4. Los eventos nuevos quedan listos para transformación posterior

---

## 3. Detección de duplicados

1. Se calcula clave de deduplicación
2. Se busca existencia previa
3. Si existe:
    - se marca como duplicate_detected
    - se conserva trazabilidad
4. Si no existe:
    - se registra en `EventDeduplicationKey`
    - continúa flujo normal

---

## 4. Reprocesamiento

1. Un evento previo falló downstream
2. Se localiza `RawEvent`
3. Se valida elegibilidad para retry
4. Se reencola
5. Se conserva historial de intentos

---

# Servicios principales

## StoreRawEvent

- persiste el payload original
- registra metadatos de recepción
- inicializa estado del evento

## DetectDuplicateEvent

- genera o valida deduplication key
- detecta eventos repetidos
- evita doble procesamiento

## AcknowledgeProviderWebhook

- devuelve respuesta técnica al proveedor
- registra resultado de recepción
- desacopla ack del resto del pipeline cuando sea posible

## ValidateIncomingSignature

- valida firma o secret del webhook
- marca el evento según resultado
- protege la entrada del sistema

## QueueRawEventForProcessing

- prepara el evento crudo para módulos posteriores
- evita que la recepción dependa de la transformación

# Diseño técnico recomendado

## Separación estricta entre ingestión y procesamiento

Debes mantener dos etapas claramente distintas:

### Etapa 1: ingestión

- recibir
- validar mínimamente
- persistir
- auditar
- responder

### Etapa 2: procesamiento

- transformar
- normalizar
- clasificar
- decidir
- accionar

Esa separación es crítica para resiliencia.

---

## Estrategia de deduplicación

La clave puede derivarse de varias combinaciones según proveedor:

- `provider + external_event_id`
- `provider + asset + event_type + occurred_at`
- checksum del payload
- combinación híbrida con ventana temporal

No todos los providers serán consistentes, así que conviene soportar reglas de deduplicación por source/provider.

---

## Persistencia del payload original

- guardar siempre el payload completo
- no sobrescribir el original
- cualquier transformación posterior debe generar nuevos artefactos, no mutar el raw
- si el payload es enorme, puede almacenarse en object storage y dejar referencia en DB

---

## Uso de colas

Este módulo debe apoyarse fuerte en colas para:

- procesamiento posterior
- retries
- batch pull
- dead-letter handling
- reprocesamiento manual o automático

# Observabilidad y operación

Este módulo debería exponer métricas como:

- eventos recibidos por proveedor
- eventos duplicados detectados
- eventos inválidos
- tiempo de ack al proveedor
- ratio de fallos por source
- backlog pendiente de procesamiento
- retries por evento
- eventos recibidos por tenant

---

# Riesgos que este módulo debe prevenir

- pérdida silenciosa de eventos
- procesamiento doble del mismo evento
- falta de trazabilidad técnica
- eventos transformados sin conservar el original
- dependencia excesiva entre recepción y procesamiento
- webhooks falsificados o malformados
- cuellos de botella en picos de tráfico
