
Es el módulo encargado de transformar los eventos crudos provenientes de múltiples proveedores en un formato interno estandarizado, consistente y desacoplado de las fuentes externas.

Este módulo define el **lenguaje canónico de eventos de SAM**, permitiendo que todos los módulos aguas abajo operen sobre una estructura uniforme, independientemente del proveedor de origen.

---

## Responsabilidad

Administrar:

- la transformación de eventos crudos (`RawEvent`) a eventos normalizados
- el mapeo de tipos de evento externos a tipos internos
- la clasificación inicial de eventos (categoría, tipo, severidad)
- la unificación semántica de eventos multi-proveedor
- el enriquecimiento básico del evento con contexto estructural
- la preparación del evento para el engine de monitoreo

---

## Qué resuelve

- estandarización de eventos heterogéneos
- compatibilidad entre múltiples proveedores
- reducción de complejidad en módulos posteriores
- eliminación de dependencia directa del formato externo
- base común para reglas, AI y decisiones
- clasificación inicial del evento
- consistencia semántica en todo el sistema

---

## Alcance del módulo

Este módulo gobierna la transición entre:

👉 datos externos crudos (inconsistentes, variables)  
👉 eventos internos estructurados (consistentes, interpretables)

Define:

- cómo se interpreta un evento externo
- cómo se traduce a un tipo interno
- qué categoría pertenece
- qué nivel de severidad tiene
- qué estructura base tendrá

No toma decisiones operativas complejas, pero sí establece el **significado del evento dentro de SAM**.

---

# Ejemplo clave

Un evento externo puede variar según proveedor:

- Samsara: `panic_button_pressed`
- Motive: `driver_alert_emergency`

Después de normalización:

category = emergency  
type = panic_button  
severity = high

👉 A partir de aquí, todo el sistema habla el mismo idioma.

# Entidades principales

## 1. NormalizedEvent

Representa el evento ya transformado al modelo interno.

### Campos sugeridos

- id
- raw_event_id
- tenant_id
- provider_id
- asset_id nullable
- driver_id nullable
- event_type_id
- event_category_id
- event_severity_id
- occurred_at
- processed_at
- payload_normalized_json
- context_json
- status
- created_at

### Notas

- Es el output principal del módulo
- Debe mantener referencia al `RawEvent` original
- Debe ser idempotente (no duplicarse por el mismo raw_event)

---

## 2. EventType

Define el tipo específico del evento.

### Campos sugeridos

- id
- code
- name
- description
- category_id
- default_severity_id
- is_active

### Ejemplos

- panic_button
- harsh_braking
- collision
- camera_obstructed
- speeding
- geofence_exit

---

## 3. EventCategory

Agrupa tipos de eventos en categorías amplias.

### Campos sugeridos

- id
- code
- name
- description

### Ejemplos

- safety
- emergency
- compliance
- operational
- maintenance

---

## 4. EventSeverity

Define el nivel de criticidad del evento.

### Campos sugeridos

- id
- code
- label
- level
- color
- response_sla_seconds

### Ejemplos

- low
- medium
- high
- critical

---

## 5. EventMappingRule

Define cómo traducir un evento externo a uno interno.

### Campos sugeridos

- id
- provider_id
- external_event_type
- external_conditions_json
- mapped_event_type_id
- mapped_category_id
- mapped_severity_id nullable
- priority
- is_active
- created_at

### Notas

- Puede haber múltiples reglas por provider
- Puede incluir condiciones adicionales (ej: velocidad > X)
- Permite evolucionar sin hardcodear lógica

---

## 6. NormalizedEventContext (opcional pero potente)

Contexto enriquecido adicional.

### Campos sugeridos

- id
- normalized_event_id
- asset_snapshot_json
- driver_snapshot_json
- location_snapshot_json
- additional_context_json

### Uso

- snapshot del estado en el momento del evento
- evita depender de datos mutables posteriores

# Reglas de negocio

## Transformación obligatoria

- todo `RawEvent` válido debe intentar transformarse en `NormalizedEvent`
- ningún módulo aguas abajo debe depender del payload crudo

## Idempotencia

- un `RawEvent` no debe generar múltiples `NormalizedEvent` duplicados
- la transformación debe ser determinística

## Trazabilidad

- todo evento normalizado debe referenciar su evento crudo
- debe poder reconstruirse el origen del evento

## Clasificación consistente

- todo evento debe tener:
    - tipo
    - categoría
    - severidad
- la severidad puede ser:
    - derivada por regla
    - heredada del tipo
    - calculada dinámicamente

## Flexibilidad

- el sistema debe permitir agregar nuevos providers sin romper el modelo
- las reglas de mapping deben ser configurables

# Flujos principales del módulo

## 1. Normalización de evento

1. Se obtiene `RawEvent`
2. Se identifica provider
3. Se extrae tipo de evento externo
4. Se busca `EventMappingRule`
5. Se determina:
    - tipo interno
    - categoría
    - severidad
6. Se construye `NormalizedEvent`
7. Se guarda resultado
8. Se marca raw_event como procesado

---

## 2. Resolución de entidades (ligera)

1. Se intenta resolver:
    - asset_id
    - driver_id
2. Se agregan referencias básicas
3. Se guarda contexto mínimo

---

## 3. Enriquecimiento básico

1. Se agregan datos relevantes:
    - ubicación
    - metadata importante
2. Se estructura `payload_normalized_json`

---

## 4. Manejo de eventos no mapeados

1. No se encuentra regla
2. Se marca como:
    - unknown_event
    - unmapped
3. Se registra para análisis
4. Puede ir a cola de revisión

---

# Servicios principales

## NormalizeRawEvent

- transforma `RawEvent` a `NormalizedEvent`
- aplica reglas de mapping
- asegura consistencia

## MapExternalEventType

- identifica tipo externo
- encuentra regla aplicable

## ResolveEventSeverity

- determina severidad final
- puede usar reglas o defaults

## AttachNormalizedContext

- agrega contexto estructurado
- prepara evento para engine

---

# Jobs

## NormalizeEventJob

- toma eventos pendientes
- ejecuta normalización
- maneja errores y reintentos

---

# Arquitectura recomendada

## Lenguaje canónico

Este módulo define el **contrato interno de eventos**:

👉 todo el sistema debe depender de `NormalizedEvent`, no de `RawEvent`

---

## Reglas configurables

- evitar hardcode
- usar `EventMappingRule`
- permitir ajustes sin redeploy

---

## Versionado (muy recomendable)

- permitir versionar reglas de mapping
- evitar romper histórico
- facilitar evolución del sistema

---

## Manejo de errores

- eventos no mapeados deben registrarse
- no deben romper el pipeline
- deben ser observables

---

# Riesgos que este módulo debe prevenir

- dependencia directa de formatos externos
- inconsistencias entre providers
- eventos mal clasificados
- pérdida de significado del evento
- lógica duplicada en múltiples módulos
- dificultad para escalar a nuevos proveedores
