Es el módulo encargado de conectar SAM con plataformas externas como telemática, video, sensores e IoT (ej. Samsara, Motive, etc.), permitiendo la ingesta, normalización y sincronización de datos hacia el sistema.

Este módulo abstrae la complejidad de múltiples proveedores y unifica la información en un modelo interno consistente para su procesamiento por el engine de monitoreo.

---

## Responsabilidad

Administrar:

- proveedores externos (telemática, video, IoT, APIs)
- conexiones por tenant hacia dichos proveedores
- credenciales y autenticación segura
- consumo de APIs externas (pull)
- recepción de eventos vía webhooks (push)
- sincronización de datos históricos y en tiempo real
- normalización de datos externos a un formato interno
- resiliencia ante fallos de integración

---

## Qué resuelve

- integración con múltiples plataformas externas
- unificación de datos heterogéneos
- gestión segura de credenciales
- ingestión de eventos en tiempo real
- sincronización periódica de datos
- desacoplamiento entre proveedores y lógica interna
- base de datos confiable para el motor de monitoreo
- manejo de errores, reintentos y fallos de APIs externas

---

## Alcance del módulo

Este módulo gobierna todo lo relacionado con la **entrada de datos externos al sistema**:

- de dónde vienen los datos
- cómo se conectan los proveedores
- cómo se autentican las integraciones
- cómo se reciben y procesan eventos
- cómo se transforman a un modelo estándar

No toma decisiones operativas (eso es del engine), pero sí garantiza que los datos lleguen **correctos, completos y normalizados**.

# Entidades principales

## 1. IntegrationProvider

Catálogo de proveedores soportados.

### Campos sugeridos

- id
- code
- name
- type (telematics, video, iot, api)
- status
- config_schema_json
- capabilities_json
- created_at

### Ejemplos

- samsara
- motive
- geotab
- lytx
- custom_api

---

## 2. TenantIntegration

Representa la conexión de un tenant con un proveedor específico.

### Campos sugeridos

- id
- tenant_id
- provider_id
- name
- status
- auth_type
- credentials_encrypted
- config_json
- last_sync_at
- created_at

### auth_type

- api_key
- oauth2
- basic_auth
- token

### Notas

- Un tenant puede tener múltiples integraciones
- Cada integración puede tener configuración específica
- Las credenciales deben estar cifradas

---

## 3. IntegrationCredential (opcional si quieres separar)

Gestión segura de credenciales.

### Campos sugeridos

- id
- tenant_integration_id
- key
- value_encrypted
- expires_at
- rotated_at

---

## 4. IntegrationSyncJob

Registro de sincronizaciones ejecutadas.

### Campos sugeridos

- id
- tenant_integration_id
- type (full, incremental, realtime)
- status
- started_at
- finished_at
- records_processed
- error_message

---

## 5. WebhookEndpoint

Endpoints registrados para recibir eventos externos.

### Campos sugeridos

- id
- tenant_integration_id
- url
- secret
- status
- last_received_at

---

## 6. WebhookEvent

Eventos recibidos desde proveedores externos.

### Campos sugeridos

- id
- tenant_id
- provider_id
- event_type
- payload_json
- received_at
- processed_at
- status
- error_message

### Campos sugeridos

# Reglas de negocio

## Integración

- cada tenant puede tener múltiples integraciones activas
- cada integración debe estar asociada a un proveedor válido
- las credenciales deben almacenarse cifradas
- se deben soportar múltiples tipos de autenticación

## Ingesta de datos

- los eventos externos deben registrarse sin pérdida
- ningún evento debe procesarse sin contexto de tenant
- los eventos deben poder reprocesarse en caso de error

## Normalización

- todos los datos externos deben transformarse a un modelo interno estándar
- el engine de SAM nunca debe depender directamente del formato del proveedor
- debe existir trazabilidad entre evento externo y evento interno

## Webhooks

- todos los webhooks deben validarse (firma, secret, etc.)
- los eventos deben almacenarse antes de procesarse
- el procesamiento debe ser asíncrono

## Sincronización

- debe soportar sync incremental y full
- deben existir reintentos automáticos
- errores no deben bloquear el sistema completo

# Flujos principales del módulo

## 1. Configuración de integración

1. Tenant selecciona proveedor
2. Se crean credenciales
3. Se valida conexión
4. Se guarda integración
5. Se inicializa sync inicial

---

## 2. Recepción de webhook

1. Proveedor envía evento
2. Se valida autenticidad
3. Se guarda `WebhookEvent`
4. Se encola procesamiento
5. Se transforma a `ExternalEvent`
6. Se envía al engine

---

## 3. Sincronización periódica

1. Job ejecuta sync
2. Se consumen APIs externas
3. Se procesan registros
4. Se normalizan datos
5. Se actualizan mappings
6. Se registra resultado

---

## 4. Procesamiento de evento

1. Evento externo llega
2. Se identifica provider
3. Se transforma a modelo interno
4. Se vincula a asset interno
5. Se envía al módulo de monitoreo
